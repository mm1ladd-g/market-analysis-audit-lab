# راهنمای مشارکت

[فارسی](CONTRIBUTING.fa.md) · [English](CONTRIBUTING.md)

از مشارکت شما در بهبود آزمایشگاه ممیزی تحلیل بازار سپاسگزاریم. تغییرها باید بررسی شواهد را آسان‌تر، پنهان‌کردن عدم‌قطعیت را دشوارتر و اجرای نرم‌افزار را امن‌تر کنند.

## پیش از آغاز

۱. Issue و Pull Request موجود را جست‌وجو کنید.

۲. برای منبع دادهٔ جدید، تغییر طرح‌واره، تغییر سیاست امتیازدهی یا پیشنهاد بزرگ معماری و رابط، ابتدا Issue باز کنید.

۳. هرگز کلید API، Cookie، رونویسی خصوصی، رسانهٔ دارای حق نشر، بستهٔ خصوصی ممیزی یا دادهٔ شخصی پیوست نکنید.

۴. در آزمون و مستندات فقط از Fixture ساختگی یا دارای مجوز روشن استفاده کنید.

۵. [روش‌شناسی](docs/fa/methodology.md)، [سیاست امنیت](SECURITY.fa.md) و [راهنمای حقوق](docs/fa/legal-and-rights.md) را بخوانید.

آسیب‌پذیری امنیتی باید مطابق [SECURITY.fa.md](SECURITY.fa.md) و خارج از Issue عمومی گزارش شود.

## محیط توسعه

</div>

```bash
cp .env.example .env
docker compose build
docker compose run --rm audit-lab python -m audit_lab.cli demo
docker compose run --rm audit-lab python -m unittest discover -s tests -v
```

<div dir="rtl" lang="fa">

در صورت نصب نسخهٔ مستند Python و وابستگی‌های قفل‌شده، اجرای محلی نیز ممکن است:

</div>

```bash
python -m unittest discover -s tests -v
python -m audit_lab.cli demo --workspace ./workspace
```

## انتظار از Pull Request

- تغییر را متمرکز نگه دارید و نتیجهٔ قابل‌مشاهده برای کاربر را توضیح دهید.
- آزمون اضافه یا به‌روز کنید؛ اصلاح Bug معمولاً باید Regression Test داشته باشد.
- حذف محافظه‌کارانه و قابلیت ردیابی شواهد را حفظ کنید.
- مستندات انگلیسی و فارسی را هم‌زمان تغییر دهید؛ هر فایل `docs/en/*.md` باید همتای `docs/fa/*.md` داشته باشد.
- در صورت لزوم نسخهٔ طرح‌واره، پرامپت، سیاست، Changelog و راهنمای مهاجرت را به‌روز کنید.
- دسترسی شبکه، گیرندهٔ داده، هزینه، Secret جدید، مجوز و اثر حریم خصوصی را مستند کنید.
- اعتبارسنجی را فقط برای عبور یک Fixture تضعیف نکنید.
- امتیاز را نرخ برد معامله، اثبات سودآوری یا تأیید صلاحیت تحلیل‌گر معرفی نکنید.

## افزودن منبع داده

مشارکت مربوط به منبع باید محل معامله، نگاشت نماد، بازه، منطقهٔ زمانی، تعدیل، شکاف، محدودیت نرخ، محدودیت مجوز و بازنشر و وضعیت دقیق یا جایگزین بودن را مستند کند. منشأ خام و هش دادهٔ عادی‌شده حفظ شود. آزمون‌ها باید از Fixture ساختگی محلی استفاده کنند و مجموعهٔ پیش‌فرض آزمون به شبکه وابسته نباشد.

## تغییر رفتار هوش مصنوعی

پرامپت، طرح‌واره، تنظیم مدل، اعتبارسنجی، Retry و حذف قطعی، اجزای نسخه‌بندی‌شدهٔ روش‌شناسی هستند. دادهٔ ارسالی را توضیح دهید، برای تزریق پرامپت و خروجی ساختاریافتهٔ خراب آزمون خصمانه اضافه کنید و ابطال Cache را نشان دهید. پاسخ مدل نباید متن مرجع رونویسی شود.

## سبک مستندات

از زبان روشن، تاریخ ISO، زمان UTC و محدودیت دقیق استفاده کنید. متن فارسی با UTF-8، `ی` و `ک` فارسی، نیم‌فاصله، Wrapper راست‌به‌چپ و بلوک کد چپ‌به‌راست نوشته شود. عنوان‌ها و مثال‌های دو زبان باید از نظر معنا یکسان باشند.

## Commit و تأیید منشأ

Commitها کوچک و توصیفی باشند. هر Commit باید امضای Developer Certificate of Origin داشته باشد:

</div>

```text
Signed-off-by: Your Name <you@example.com>
```

<div dir="rtl" lang="fa">

از `git commit -s` استفاده کنید. با Sign-off، متن [DCO](DCO) را تأیید می‌کنید.

## بازبینی و ادغام

نگه‌دارندگان درستی، روش‌شناسی، حقوق و حریم خصوصی، امنیت، آزمون، برابری مستندات و نگه‌داری‌پذیری را بررسی می‌کنند. پذیرش تضمین‌شده نیست. تغییر مهم روش‌شناسی باید پیش از ادغام، Issue و سابقهٔ تصمیم داشته باشد. تغییرِ افشاکنندهٔ دادهٔ خصوصی، ایجادکنندهٔ خطر حقوقی یا تضعیف‌کنندهٔ انصاف شواهد ممکن است بسته شود.
