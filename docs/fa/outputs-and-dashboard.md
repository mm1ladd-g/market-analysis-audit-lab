# خروجی و داشبورد

[فارسی](outputs-and-dashboard.md) · [English](../en/outputs-and-dashboard.md) · [مستندات](index.md)

مرحلهٔ `report` دادهٔ ارائه را از تحلیل نهایی مشتق می‌کند و نباید تصمیم ادعا یا امتیاز را عوض کند. داشبورد و PDF نمای داده‌اند، نه منبع مستقل حقیقت.

کنار هر تیتر، بازه، تعداد کشف/ورود/حذف، ادعای کل/شمرده، توزیع نتیجه، شرط فعال‌نشده، دادهٔ ناکافی، دامنه، Provider و Proxy، نسخهٔ مدل/روش، محدودیت، رابطه و مسیر اصلاح را نشان دهید. درصد را «هم‌راستایی سناریو و پیامد در ادعاهای فعال و قابل‌راستی‌آزمایی» بنامید و مخرج را نزدیک آن بگذارید.

نمای عمومی باید کمترین شاهد لازم، متن Escape‌شده، بدون مسیر محلی، رونویسی خام، Cache یا API حساس بدون احراز هویت باشد. دارایی بصری باید در `ASSET_LICENSES.yml` ثبت شود. تغییر زبان باید جهت، متن، Alt، تاریخ/عدد و Report را عوض کند؛ Hash، Chart و Symbol چپ‌به‌راست می‌مانند.

فایل `reports/dashboard_data.json` اثر داخلی ارائه است و ممکن است جزئیات خصوصی لازم برای PDF دوزبانه داشته باشد؛ این فایل قرارداد API عمومی نیست. صفحه HTML و `/api/dashboard` از DTO تجمیعی و Allowlist‌شده استفاده می‌کنند. وقتی `PUBLIC_CLAIM_LEDGER=false` است، متن/شاهد ادعا، استدلال مدل، URL ویدئو، مسیر محلی و خطای Provider هرگز برگردانده نمی‌شود. حالت خصوصی با «پیش‌نمایش خصوصی» و حالت عمومی بدون پذیرش معتبر با «در انتظار بازبینی انسانی» مشخص می‌شود و دانلود Report/دفتر بسته می‌ماند.

پس از پذیرش بازبینی شواهد، داشبورد و PDF بازتولیدشده و هر دفتر ادعای عمومیِ اختیاری را بررسی و سپس `review publication-accept` را اجرا کنید. `finalize` عمومی این ایست بازرسی ارائه را الزامی می‌کند، پوشه و ZIP نهایی کامل را راستی‌آزمایی می‌کند و فقط پس از آن Manifest انتشاری را فعال می‌کند که به آثار پذیرفته‌شده متصل به هش است. داشبورد و هر دانلود، فایل قابل‌ارائه را با هش فعال‌شده مقایسه می‌کنند. فایل حذف‌شده، بازتولیدشده یا جایگزین‌شده به‌جای به‌ارث‌بردن پذیرش قبلی باعث بسته‌شدن امن دسترسی می‌شود.

## ساخت PDF دوزبانه

پس از مرحلهٔ `report`، سند فارسی‌اول و انگلیسی‌دوم را بسازید و بررسی کنید، آثار دقیق ارائه را بپذیرید و سپس مرحلهٔ نهایی‌سازی را اجرا کنید تا PDF وارد بستهٔ قابل‌راستی‌آزمایی با هش شود:

</div>

```bash
docker compose run --rm audit-lab python scripts/generate_audit_report.py --workspace /workspace
# پیش از ادامه، dashboard_data.json، فایل PDF و دفتر عمومی اختیاری را بررسی کنید.
docker compose run --rm audit-lab python -m audit_lab.cli review publication-accept \
  --reviewer "نام بازبین" --notes "داشبورد، PDF و دفتر عمومی اختیاری بررسی شد"
docker compose run --rm audit-lab python -m audit_lab.cli finalize
docker compose run --rm audit-lab python -m audit_lab.cli verify-final
```

<div dir="rtl" lang="fa">

خروجی ثابت در `workspace/reports/audit-report.pdf` نوشته می‌شود. این سند فقط نمایی از `dashboard_data.json` است و نتیجهٔ ممیزی را دوباره محاسبه یا بهتر نمی‌کند.

برای انتشار واقعی، پذیرش فعلیِ بازبینی شواهد را ثبت، `report` را دوباره تولید، PDF فارسی‌اول/انگلیسی‌دوم و هر دفتر عمومی اختیاری را بسازید و بررسی کنید، `review publication-accept` را ثبت کنید و سپس بدون تغییر اثر پذیرفته‌شده Finalize و Verify کنید. تغییر مجموعه، پیامد یا امتیاز، پذیرش شواهد را قدیمی می‌کند. اجرای دوبارهٔ `report`، بازتولید PDF، جایگزینی دفتر اختیاری یا تغییر هر یک از دفترهای بازبینی، مجوز و Manifest انتشار فعال‌شده را نامعتبر می‌کند؛ پیش از ارائهٔ داشبورد یا دانلودها، `review publication-accept` و `finalize` را تکرار کنید.
