## پیش از اجرا

اپراتور، هدف، منابع مجاز، مرز تاریخ، منطقهٔ زمانی، دسته‌ها، ارائه‌دهندگان، تعداد موردانتظار ویدئو، سقف بودجه، مسئول بازبینی، مدت نگهداری، `AUDIT_RELATIONSHIP_DISCLOSURE` و `CORRECTION_CONTACT` را ثبت کنید. پیش از استفادهٔ دوباره از فضای کاری موجود، از آن نسخهٔ پشتیبان بگیرید.

</div>

```bash
python -m audit_lab.cli doctor
python -m audit_lab.cli smoke
python -m audit_lab.cli status
```

## مراحل

۱. `collect` فرادادهٔ منبع مجاز و زیرنویس‌های ترجیحی را ثبت می‌کند. این مرحله باید ثابت کند پیمایش از `START_DATE` عبور کرده یا منبع تمام شده است.

۲. `transcribe` در صورت فعال‌شدن صریح مسیر جایگزین، می‌تواند از صوت مجاز برای ویدئوی بدون زیرنویس رونویسی زمان‌دار بسازد؛ صوت خام به‌طور پیش‌فرض حذف می‌شود.

۳. `manifest` سیاست کانال/تاریخ/دسته/زیرنویس را دوباره کنترل، رونویسی را عادی‌سازی، حذف‌ها و فهرست‌های SHA-256 را ثبت و بستهٔ نسخه‌بندی‌شدهٔ مجموعه را ایجاد می‌کند.

۴. `verify` هش‌ها را دوباره محاسبه و ZIP را آزمایش می‌کند. در صورت شکست متوقف شوید.

۵. `extract-claims` ادعاهای ابطال‌پذیر را همراه محدودهٔ خطوط منبع انتخاب می‌کند. با `--limit 1` اجرا و بررسی کنید، سپس ادامه دهید. `--force` دوباره هزینه می‌کند و شواهد تازهٔ مدل می‌سازد.

۶. `fetch-outcomes` دارایی‌های پشتیبانی‌شده را حل می‌کند، ردیف‌های بازار را می‌گیرد یا وارد می‌کند، بازار/فاصلهٔ زمانی/نماد جایگزین را ثبت می‌کند و پنجره‌های ۲۴ و ۴۸ ساعت را می‌سازد.

۷. `score` ابتدا حذف‌های قطعی را اعمال می‌کند و فقط برای تفسیر باقی‌مانده از مدل تنظیم‌شده استفاده می‌کند. پیش از اجرای کامل، یک ویدئو را بیازمایید.

۸. `report` دادهٔ JSON ارائه را مشتق می‌کند و نباید امتیازها را دوباره محاسبه کند.

۹. برای انتشار عمومی، بازبینی شواهد را بپذیرید، `report` را دوباره اجرا کنید، PDF و دفتر عمومی اختیاری را بسازید و بررسی کنید و سپس `review publication-accept` را اجرا کنید تا بایت‌های دقیق فایل‌های ارائه ثبت شوند.

۱۰. `finalize` اجزای الزامی، روش‌شناسی، فایل‌های بازتولید و فهرست‌های هش را در یک بستهٔ کامل کپی می‌کند. در حالت عمومی هر دو ایست بازبینی معتبر را می‌خواهد، پوشه و ZIP را راستی‌آزمایی می‌کند و فقط سپس مانیفست انتشار متصل به هش را به‌طور یکپارچه فعال می‌کند.

۱۱. `verify-final` پوشه و ZIP نهایی را مستقلاً باز و هش می‌کند.

تنها پس از شناخت هر مرحله از `python -m audit_lab.cli run --stop-after STAGE` استفاده کنید؛ این دستور جای ایست‌های بازبینی را نمی‌گیرد.

داشبورد عمومی، PDF و دفتر ادعای صریحاً فعال‌شده تا زمانی که هش فعلی آن‌ها با مانیفست انتشار فعال منطبق نباشد به‌صورت امن بسته می‌مانند. پس از اجرای دوبارهٔ `report`، بازتولید یا جایگزینی PDF، تغییر هر دفتر بازبینی یا جایگزینی دفتر اختیاری، پیش از بازگرداندن دسترسی عمومی `review publication-accept` و `finalize` را تکرار کنید.

## بازیابی از شکست

برای عبور یک مرحله، شواهد را حذف نکنید. مشکل تنظیمات یا ارائه‌دهنده را رفع کنید، log شکست را نگه دارید و کوچک‌ترین مرحلهٔ تحت‌تأثیر را دوباره اجرا کنید. کلیدهای حافظهٔ نهان پس از تغییر مرتبط در رونویسی، پرامپت، طرح‌واره، سیاست، مدل یا پیامد مانع استفادهٔ دوباره می‌شوند. ناسازگاری هش یعنی فایل تا زمان بازسازی یا بازیابی قابل‌اعتماد نیست.

## ایست‌های انسانی

کامل‌بودن گردآوری، حذف‌ها، دسته‌های ناشناخته، هر نگاشت تازهٔ ارائه‌دهنده، یک استخراج برای هر زبان یا سبک محتوا، شرط‌های فعال‌شدن، مخرج شمرده‌شده، تصمیم حساس به نماد جایگزین، متن نتیجه، حقوق، حریم خصوصی و گزیده‌های عمومی را بررسی کنید. هویت بازبین و `CORRECTION_CONTACT` را بدون اطلاعات محرمانه ثبت کنید.
