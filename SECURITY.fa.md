# سیاست امنیت

[فارسی](SECURITY.fa.md) · [English](SECURITY.md)

## نسخه‌های پشتیبانی‌شده

تا پیش از نسخهٔ ۱٫۰، اصلاحات امنیتی روی جدیدترین انتشار برچسب‌خورده و شاخهٔ پیش‌فرض اعمال می‌شوند. ممکن است نسخه‌های قدیمی‌تر پشتیبانی نشوند. هر استثنا در یادداشت انتشار اعلام خواهد شد.

## گزارش خصوصی آسیب‌پذیری

از گزینهٔ **Report a vulnerability** در بخش Security مخزن GitHub استفاده کنید. جزئیات سوءاستفاده را در Issue، Discussion، Pull Request یا فایل گزارش عمومی ننویسید. اگر گزارش خصوصی فعال نبود، فقط یک Issue با متن «Private security contact needed» باز کنید تا نگه‌دارنده یک مسیر خصوصی ایجاد کند.

در صورت امکان و بدون افشای دادهٔ حساس، این موارد را ارائه دهید:

- نسخه، Commit، Digest تصویر و پلتفرم آسیب‌دیده؛
- نمونهٔ حداقلی بدون کلید واقعی یا دادهٔ شخصی شخص ثالث؛
- رفتار موردانتظار و مشاهده‌شده؛
- اثر احتمالی و نیاز یا عدم‌نیاز به دسترسی شبکه یا ورودی غیرقابل‌اعتماد؛
- راهکار پیشنهادی، در صورت آگاهی.

نگه‌دارندگان در حد توان گزارش را تأیید، اثر را بررسی و افشا و اصلاح هماهنگ را مدیریت می‌کنند. پروژه زمان پاسخ تضمین‌شده‌ای وعده نمی‌دهد.

## اسرار و اعتبارنامه‌ها

- فایل `.env`، کلید API، توکن OAuth، Cookie، اعتبارنامهٔ Tunnel و رمز منبع داده را وارد Git نکنید.
- کلید را در Build Argument، متغیر `ENV` تصویر، آرگومان خط فرمان، پیوست Issue یا بستهٔ ممیزی قرار ندهید.
- برای استقرار مشترک از مدیر اسرار یا تزریق فقط هنگام اجرا استفاده کنید.
- در صورت افشا، کلید را فوراً لغو یا تعویض کنید. حذف از آخرین Commit کافی نیست؛ تمام تاریخچه، Release، لایه‌های تصویر، Cache و Log را اسکن کنید.
- CLI فقط وجود کلید را گزارش می‌کند و نباید مقدار یا نمایش برگشت‌پذیر آن را چاپ کند.

## مدل تهدید

تمام ورودی‌های بیرونی غیرقابل‌اعتماد هستند: فرادادهٔ کانال، عنوان، توضیح، زیرنویس، رونویسی، CSV، اعضای آرشیو، خروجی مدل، URL و HTML نهایی.

خطرهای مهم:

- تزریق پرامپت در رونویسی یا فراداده؛
- پیمایش مسیر، Symlink، بمب ZIP، بمب بازکردن فشرده و فایل بیش‌ازحد بزرگ؛
- تزریق فرمول در CSV؛
- XSS در عنوان، شاهد، توضیح یا متن مدل؛
- SSRF و تغییر مسیر ناخواسته از Endpoint قابل‌تنظیم؛
- محروم‌سازی از سرویس با کانال بزرگ، بازهٔ طولانی، Retry یا Concurrency بالا؛
- به‌روزرسانی آلودهٔ وابستگی یا تصویر؛
- انتشار محتوای دارای حق نشر، دادهٔ شخصی یا اثر API محرمانه؛
- قرار دادن API ادعا یا گزارش بدون احراز هویت روی اینترنت.

مدل، شواهد را داده می‌داند و نه دستور قابل‌اعتماد. خروجی ساختاریافته، کپی متن مرجع، طرح‌وارهٔ سخت‌گیرانه، محدودیت اندازه، Escape کردن HTML، استخراج امن آرشیو و بازبینی انسان لایه‌های دفاعی هستند؛ هیچ‌کدام به‌تنهایی کافی نیست.

## خط پایهٔ استقرار

- اتصال پیش‌فرض به `127.0.0.1` باشد.
- پیش از دسترسی عمومی، TLS، احراز هویت، مجوزدهی، محدودیت درخواست، Header امن، حذف اسرار از Log، پشتیبان‌گیری و بازبینی انتشار اضافه کنید.
- Container را تا حد امکان با کاربر غیر `root` و فایل‌سیستم ریشهٔ فقط‌خواندنی اجرا کنید و فقط Workspace لازم را Mount کنید.
- Docker، فایل‌سیستم میزبان، `.env`، Cache، شواهد خام و فرادادهٔ Build را منتشر نکنید.
- Egress را به منابع مستند محدود و Redirectها را بررسی کنید.
- وابستگی، GitHub Action و تصویر پایه را Pin و اسکن کنید و برای Release، SBOM بسازید.
- مسیرهای `/api/claims`، شاهدهای رونویسی و گزارش‌ها تا زمان تأیید انتشار، حساس هستند.

## استثنای بازبینی‌شدهٔ اسکن امنیتی

استثناهای اسکن امنیتی در فایل `.grype.yaml` محدود، وابسته به نسخه و قابل
بازبینی نگه داشته می‌شوند. در تاریخ ۲۰۲۶-۰۷-۱۶ تنها استثنا
`CVE-2026-15308` برای باینری CPython 3.14.6 است. این هشدار به حملهٔ منع خدمت
پردازنده در `html.parser.HTMLParser` مربوط است؛ این برنامه آن parser را import
نمی‌کند و هیچ مسیر وبی برای پردازش HTML کنترل‌شده توسط کاربر ندارد. Grype رفع
مشکل را فقط در Python 3.15.0 نشان می‌دهد که هنوز نسخهٔ پایدار نیست. این استثنا
باید با انتشار نخستین تصویر پایدار و اصلاح‌شدهٔ CPython حذف شود و حداکثر تا
۲۰۲۶-۱۰-۳۱ دوباره بازبینی شود. هیچ بسته، نسخهٔ Python، آسیب‌پذیری یا سطح شدت
دیگری با این استثنا نادیده گرفته نمی‌شود.

## کنترل پیش از انتشار

آزمون‌ها، اسکن اسرار در کل تاریخچه، بررسی مجوز و وابستگی، تحلیل ایستا، اسکن تصویر، ساخت SBOM و بررسی `docker history` را اجرا کنید. یک Clone پاک باید بدون هیچ کلید یا اثر خصوصی، نمونهٔ ساختگی را بسازد و اجرا کند.

دروازهٔ انتشار Container برای هر آسیب‌پذیری سطح بالا یا بحرانی که اسکنر برای
آن اصلاح قابل‌دسترسی معرفی کند شکست می‌خورد. یافته‌های بدون اصلاح شناخته‌شده
همراه SBOM گزارش و برای بازبینی نگه‌داری می‌شوند، اما به‌صورت خودکار انتشار را
متوقف نمی‌کنند؛ اسکنر بالادستی و توزیع ممکن است دربارهٔ انطباق بسته اختلاف
داشته باشند و اصلاحی که وجود ندارد را نمی‌توان به‌شکل بازتولیدپذیر اعمال کرد.
هر استثنای محدود به ارزیابی دسترسی‌پذیری مستندشده، مطابق بخش بالا، نیاز دارد.

کنترل امنیتی خطر را کاهش می‌دهد، اما نرم‌افزار را برای کاربرد مالی، حیاتی یا قانون‌گذاری‌شده تأیید نمی‌کند.
